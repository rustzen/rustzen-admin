# Deployment Guide

This repository builds and deploys one complete `rz` release. It does not
support Server/Web split deployment, module-only releases, dynamic activation,
long-lived mixed versions, or zero-downtime topology.

## Local commands

```bash
just check
just verify-processes
just build-native
just build
```

All commands run locally. `just build-native` builds the current machine's
optimized binary after a Bun Web build. `just build` performs the local Docker
Linux build, signs `target/rz/rz-<version>-x86_64`, and writes installation
configuration and four service units under `target/rz/`.

The Web package manager is Bun 1.3.14. `apps/web/bun.lock` is the only frontend
lockfile.

## Artifact contract

- File name: `rz-<version>-<x86_64|aarch64>`.
- ELF machine architecture must match the declared architecture.
- The binary contains `RUSTZEN_RELEASE_MARKER` with `artifact=rz`.
- Production uploads require the appended Ed25519 marker generated by:

```bash
bun scripts/deploy-sign.mjs sign-release \
  --file target/rz/rz-<version>-<arch> \
  --version <version> \
  --arch <arch>
```

The signature covers component `release`, version, architecture, and the
SHA-256 of the complete binary before the signature marker. Web resources are
already embedded and are not signed or uploaded separately.

The signing command never creates or rotates a private key. It resolves the
existing Ed25519 private key in this order: `RUSTZEN_DEPLOY_SIGN_KEY_FILE`,
`RUSTZEN_DEPLOY_SIGN_KEY`, the configured/default iCloud
`rustzen/secrets` directory, then the established local Admin key. It fails
closed when none is available, so the deployed public verify key cannot be
silently replaced.

## Installation

`deploy/setup-layout.sh` installs one versioned binary, creates `bin/rz`, keeps
an existing `config/rz.env`, and installs:

- `rz-monitor.service`
- `rz-insights.service`
- `rz-reports.service`
- `rz-admin.service`

Production must replace both `RUSTZEN_JWT_SECRET` and `RUSTZEN_IPC_TOKEN` and
must retain `RUSTZEN_DEPLOY_SIGNATURE_REQUIRED=true` with the matching public
verify key. The installer enables services but does not start them with release
placeholder secrets.

## Apply and rollback

Only `owner` may submit apply, rollback, cleanup, or other release mutations.
Admin and Viewer can list release state only.

Apply starts a fixed transient systemd unit named `rz-update-<numeric-id>`.
The unit has a bounded three-attempt restart policy for an abruptly terminated
Update Worker. A normal health-gate failure rolls back and exits without
retrying the rejected Release.
The worker:

1. revalidates file hash, ELF architecture, embedded marker, and signature;
2. creates consistent online `VACUUM INTO` backups for `admin.db`, `monitor.db`,
   `insights.db`, and `reports.db` without changing the current symlink;
3. atomically switches `/opt/rz/bin/rz` once;
4. restarts Monitor, Insights, Reports, then Admin one at a time, requiring each
   health probe before the next restart;
5. marks the release current only after all probes pass.

Before switching the link, the worker atomically writes
`data/update-state.json`. After an interruption, the bounded restart first
restores the previous link and each database whose owning process entered the
restart sequence, verifies those previous processes, removes the journal, and
only then retries the requested Release. This prevents two Release records from
remaining Current.

On any failure it stops the sequence, restores the previous symlink, restores
the databases owned by processes that entered the restart sequence, and starts
those processes on the previous release. Backups and failed candidates remain
for diagnosis; cleanup is a separate owner action.

## Monitor Agent rollout

`rz monitor agent` uses the same source and version line but runs on a managed
node under `deploy/rz-monitor-agent.service`. A rollout is inactive unless all
`RUSTZEN_MONITOR_AGENT_RELEASE_*` fields and a rollout stage are configured.
Use `canary` with explicit `RUSTZEN_MONITOR_AGENT_CANARY_IDS` first; after
verification, use `batch` with a bounded
`RUSTZEN_MONITOR_AGENT_BATCH_PERCENT`. The Controller will not issue a directive
when OS, architecture, protocol, current version, or free-space checks fail.

The Agent accepts only the HMAC-signed fixed update directive. It enforces a
64 MiB limit, exact size and SHA-256, ELF architecture, a fixed candidate
self-test, same-directory atomic replacement, and previous-binary retention.
The first successful heartbeat confirms the update; three consecutive failed
heartbeats restore the previous binary and request a supervisor restart.

## Local gates

- `cargo test --workspace`
- `bun run vp lint`
- `bun run build`
- `just build-native`
- `just verify-processes`
- `git diff --check`

`just verify-processes` terminates each Worker independently and injects
corruption into Admin, Monitor, Insights, and Reports databases one at a time.
It verifies that unrelated processes stay healthy and restoring each database
recovers its owning process.
